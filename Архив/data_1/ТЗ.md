## Модуль `data` – Техническое задание

### 1. Назначение модуля

Модуль отвечает за:
- Взаимодействие с REST API чат-сервера (https://faerytea.name/).
- Локальное хранение аутентификационных данных (токен, имя пользователя).
- Предоставление единого репозитория для остальных модулей (ViewModel) через набор `suspend`-функций.

Модуль **не зависит** от Android UI, не содержит классов `Activity`/`Fragment`/`Composable`.  
Доступ к `Context` допускается только внутри `storage` (через `SharedPreferences` или `DataStore`).

### 2. Структура пакетов и классы

```
data/
├── api/
│   ├── ChatApiService.kt          // интерфейс Retrofit
│   ├── dto/                       // DTO для сериализации
│   │   ├── LoginRequest.kt
│   │   ├── LoginResponse.kt
│   │   ├── MessageDto.kt
│   │   └── MessageData.kt
│   └── NetworkModule.kt           // предоставляет экземпляр ChatApiService (DI)
├── repository/
│   ├── ChatRepository.kt          // главный репозиторий
│   └── UnauthorizedException.kt   // кастомное исключение для 401
├── storage/
│   └── AuthStorage.kt             // хранение токена и имени пользователя
└── utils/
    └── Constants.kt               // константы (BASE_URL, таймауты и т.п.)
```

### 3. Компонент `api` – сетевая часть

#### 3.1. Базовые настройки сети

- **Base URL**: `https://faerytea.name/` (константа `BASE_URL`).
- **Формат обмена**: JSON (кроме `multipart` для картинок).
- **Библиотека**: Retrofit 2 + OkHttp 3 + сериализатор (Moshi или kotlinx.serialization).
- **Логгирование** (опционально): можно добавить `HttpLoggingInterceptor` на уровне `debug`.
- **Таймауты**: соединение – 15 секунд, чтение – 30 секунд.

#### 3.2. Интерфейс `ChatApiService`

Все функции – `suspend`. Определения:

```kotlin
interface ChatApiService {
    @POST("login")
    @Headers("Content-Type: application/json")
    suspend fun login(@Body request: LoginRequest): LoginResponse

    @GET("channels")
    suspend fun getChannels(@Header("X-Auth-Token") token: String): List<String>

    @GET("channel/{channelName}")
    suspend fun getMessages(
        @Header("X-Auth-Token") token: String,
        @Path("channelName") channelName: String,
        @Query("limit") limit: Int = 20,
        @Query("lastKnownId") lastKnownId: Int = 0
    ): List<MessageDto>

    @POST("messages")
    @Headers("Content-Type: application/json")
    suspend fun sendTextMessage(
        @Header("X-Auth-Token") token: String,
        @Body message: MessageDto
    ): Response<Unit>   // возвращает только код 200, тело может быть пустым или id

    @Multipart
    @POST("messages")
    suspend fun sendImageMessage(
        @Header("X-Auth-Token") token: String,
        @Part("msg") msg: MultipartBody.Part,   // JSON-строка MessageDto
        @Part picture: MultipartBody.Part       // файл с именем "picture"
    ): Response<Unit>

    @POST("logout")
    suspend fun logout(@Header("X-Auth-Token") token: String): Response<Unit>
}
```

**Примечания по `sendTextMessage` / `sendImageMessage`**:
- Сервер возвращает `200 OK` с телом, содержащим `id` сообщения. Можно игнорировать тело, поэтому возвращаем `Response<Unit>`.
- Для `sendImageMessage`: поле `msg` должно быть сформировано как `MultipartBody.Part` из JSON-строки (например, `{"from":"user","to":"1@channel","data":{"Text":{"text":"описание"}}}`).
- Не забываем указать правильный `Content-Disposition` для `picture`.

#### 3.3. DTO-классы

Все DTO – `data class` с неизменяемыми полями (`val`). Сериализуются стандартно (аннотации зависят от выбранного сериализатора).

**LoginRequest**
```kotlin
data class LoginRequest(
    val name: String,
    val pwd: String
)
```

**LoginResponse**
```kotlin
data class LoginResponse(
    val token: String
)
```

**MessageDto**
```kotlin
data class MessageDto(
    val id: String? = null,          // игнорируется в запросах
    val from: String,                // обязательное
    val to: String = "1@channel",    // по умолчанию общий канал
    val data: MessageData,
    val time: Long? = null           // опционально, UNIX time
)
```

**MessageData** – sealed class (или иерархия с `@JsonTypeInfo`):
```kotlin
sealed class MessageData {
    data class TextMessage(val text: String) : MessageData()
    data class ImageMessage(val link: String? = null) : MessageData()   // link – путь к картинке, например "pic.jpg"
}
```
*При сериализации:*
- `{"Text":{"text":"hello"}}`
- `{"Image":{"link":"pic.jpg"}}`

#### 3.4. Обработка ошибок HTTP

Retrofit автоматически выбрасывает `HttpException` для кодов 4xx/5xx. Это нормально.  
`ChatApiService` не перехватывает эти исключения – они обрабатываются выше (в репозитории).

### 4. Компонент `storage` – локальное хранилище

#### 4.1. Класс `AuthStorage`

Используется `SharedPreferences` (или `Preferences DataStore`).  
Конструктор принимает `Context` (через `ApplicationContext`, чтобы избежать утечек).

Методы:

```kotlin
class AuthStorage(context: Context) {
    suspend fun saveToken(token: String, userName: String)
    fun getToken(): String?          // можно синхронно (SharedPreferences – быстрый)
    fun getUserName(): String?
    suspend fun clear()
}
```

- `saveToken` и `clear` – `suspend` только если используется DataStore; при SharedPreferences они могут быть обычными функциями. Для единообразия допустимы обычные (non-suspend) функции.
- Храним две строки: `auth_token` и `user_name`.

**Важно**: никакой логики, кроме чтения/записи.

### 5. Компонент `repository` – репозиторий

#### 5.1. Класс `ChatRepository`

Конструктор принимает:
- `api: ChatApiService`
- `authStorage: AuthStorage`

Все публичные функции – `suspend`. Возвращают `Result<T>` для явной обработки успеха/ошибки.  
**Исключение**: `getImageUrl` – не suspend, возвращает готовый URL.

#### 5.2. Определения функций

```kotlin
class ChatRepository(
    private val api: ChatApiService,
    private val authStorage: AuthStorage
) {
    suspend fun login(name: String, password: String): Result<Unit>
    suspend fun logout(): Result<Unit>
    suspend fun getChannels(): Result<List<String>>
    suspend fun getMessages(channelName: String, limit: Int = 20, lastKnownId: Int = 0): Result<List<MessageDto>>
    suspend fun sendTextMessage(channelName: String, text: String): Result<Unit>
    suspend fun sendImageMessage(channelName: String, imageFile: File, caption: String? = null): Result<Unit>
    fun getImageUrl(path: String, thumb: Boolean): String
}
```

#### 5.3. Реализация требований

**5.3.1. `login`**
- Вызвать `api.login(LoginRequest(name, pwd))`.
- При успехе (получен токен) вызвать `authStorage.saveToken(token, name)` и вернуть `Result.success(Unit)`.
- При ошибке (сетевой или `HttpException`) вернуть `Result.failure(exception)`.
- Не нужно сохранять токен при ошибке.

**5.3.2. `logout`**
- Если `authStorage.getToken() != null`, вызвать `api.logout(token)` (игнорируя возможные ошибки сети – для надёжности).
- Затем `authStorage.clear()`.
- Вернуть `Result.success(Unit)` (даже если сетевой вызов провалился, локально данные очищены).

**5.3.3. `getChannels`, `getMessages`, `sendTextMessage`, `sendImageMessage`**
- Перед вызовом API получить токен: `val token = authStorage.getToken()`.
- Если `token == null` – вернуть `Result.failure(UnauthorizedException())`.
- Выполнить запрос.
- При `HttpException` с кодом `401` – **только** вернуть `Result.failure(UnauthorizedException())`, не очищая токен. Решение о повторном логине принимает ViewModel.
- При любой другой ошибке – вернуть `Result.failure(exception)`.
- При успехе – `Result.success(data)`.

**5.3.4. `sendImageMessage`**
- Параметры: `channelName`, `imageFile` (File), `caption` (опциональный текст).
- Сформировать `MessageDto`:
  - `from = authStorage.getUserName()` (если null – вернуть `UnauthorizedException`).
  - `to = channelName`
  - `data` = если caption не пуст, то `TextMessage(caption)`, иначе `ImageMessage()` (без `link`, пусть сервер заполнит).
- Сериализовать этот DTO в JSON-строку (использовать тот же сериализатор, что и Retrofit).
- Создать `MultipartBody.Part` для `msg`:
  ```kotlin
  val msgPart = MultipartBody.Part.createFormData("msg", null, jsonString.toRequestBody(MediaType.parse("application/json")))
  ```
- Создать `MultipartBody.Part` для `picture`:
  ```kotlin
  val picturePart = MultipartBody.Part.createFormData("picture", imageFile.name, imageFile.asRequestBody(MediaType.parse("image/*")))
  ```
- Вызвать `api.sendImageMessage(token, msgPart, picturePart)`.

**5.3.5. `getImageUrl`**
- `thumb = true` → `"https://faerytea.name/thumb/$path"`
- `thumb = false` → `"https://faerytea.name/img/$path"`
- Просто строит строку, не выполняет сетевых вызовов.

#### 5.4. Кастомное исключение `UnauthorizedException`

```kotlin
class UnauthorizedException : Exception("Authentication token is missing or invalid")
```

### 6. Требования к коду

- **Никаких `var` и мутабельных коллекций** в публичных API (все `val`, возвращаемые списки – неизменяемые).
- **Никаких варнингов** в Android Studio (жёлтые подчёркивания): все ресурсы строк вынесены в `strings.xml` **только для UI**, но в модуле `data` строки-константы определяются внутри класса/companion object или в `Constants.kt`.
- **Хардкод строк** (URL, имена полей) допускается только как константы с говорящими именами.
- **Поведение при повороте** не касается модуля `data` (это забота ViewModel и UI).
- **Не блокировать поток**: все функции, делающие сетевые вызовы или обращение к DataStore, должны быть `suspend`. `getToken()` может быть синхронной (SharedPreferences).
- **Пакет приложения** – не должен начинаться с `com.example`. Рекомендуется, например, `ru.uni.chat.data`.
- **Сборка**: модуль должен компилироваться без ошибок и не падать при корректных входных данных.

### 8. Пример использования в ViewModel (для пояснения – не часть ТЗ)

```kotlin
// В ViewModel (ui-модуль)
viewModelScope.launch {
    val result = repository.getChannels()
    result.onSuccess { channels ->
        _channels.value = channels
    }.onFailure { error ->
        if (error is UnauthorizedException) {
            _events.emit(NavigateToLogin)
        } else {
            _events.emit(ShowError("Ошибка загрузки чатов"))
        }
    }
}
```

### 9. Критерии приёмки модуля `data`

- Все описанные классы присутствуют и компилируются.
- Логин сохраняет токен и имя пользователя.
- Запросы списка каналов и сообщений работают с корректным токеном.
- Отправка текстового сообщения работает.
- Отправка картинки через multipart работает.
- При отсутствии токена или `401` репозиторий возвращает `UnauthorizedException`, не очищая хранилище.
- Выход очищает токен и вызывает `/logout`.
- Нет предупреждений линтера, нет `var` в публичных API, нет `com.example` в package.
