## Модуль `data`

Назначение: инкапсулирует работу с сервером (REST API) и локальным хранилищем (токен, имя пользователя). Предоставляет репозиторий с чистыми suspend-функциями для остальных модулей.

### Основные компоненты

#### 1. `data.api` — сетевая часть

* `ChatApiService` (интерфейс Retrofit):
    * `POST /login` – получить токен (тело: `{ "name": String, "pwd": String }`, ответ: `{ "token": String }`).
    * `GET /channels` – получить список каналов (массив строк). Заголовок: `X-Auth-Token`.
    * `GET /channel/{channelName}?limit=20&lastKnownId=0` – получить сообщения канала. Заголовок: `X-Auth-Token`. Возвращает список `MessageDto`.
    * `POST /messages (JSON)` – отправить текстовое сообщение. Заголовок: `X-Auth-Token`. Тело – `MessageDto`.
    * `POST /messages (multipart)` – отправить сообщение с картинкой. Поля: `msg` (JSON-строка) и `picture` (файл).
    * `POST /logout` – забыть токен на сервере. Заголовок: `X-Auth-Token`.
* DTO-классы (сериализуемые):
    * `LoginRequest(name: String, pwd: String)`
    * `LoginResponse(token: String)`
    * `MessageDto`:
        * `id: String?`
        * `from: String`
        * `to: String = "1@channel"`
        * `data: MessageData`
        * `time: Long?`
    * `MessageData`:
        * `TextMessage(text: String)`
        * `ImageMessage(link: String?)` (`link` – путь к картинке, например `"pic.jpg"`)
* `ChatApiService` должен корректно обрабатывать ошибки HTTP (401, 404, 500 и т.д.), выбрасывая `HttpException` из Retrofit.

#### 2. `data.repository` — репозиторий

Класс `ChatRepository` (все функции `suspend`, возвращают `Result<T>` или бросают исключения на усмотрение, но лучше `Result` для явной обработки ошибок в `ViewModel`).

**Функции:**

* `suspend fun login(name: String, password: String): Result<Unit>` — вызывает `/login`, при успехе сохраняет `token` и `userName` в `AuthStorage`.
* `suspend fun logout(): Result<Unit>` — вызывает `/logout` (если есть токен) и очищает `AuthStorage`.
* `suspend fun getChannels(): Result<List<String>>` — использует сохранённый токен.
* `suspend fun getMessages(channelName: String, limit: Int = 20, lastKnownId: Int = 0): Result<List<MessageDto>>`
* `suspend fun sendTextMessage(channelName: String, text: String): Result<Unit>`
* `suspend fun sendImageMessage(channelName: String, imageFile: MultipartBody.Part): Result<Unit>`
* `fun getImageUrl(path: String, thumb: Boolean): String` — возвращает `https://faerytea.name/thumb/{path}` или `/img/{path}`. Не `suspend`.

**Обработка 401:**
Если при любом запросе (кроме логина) приходит `401 Unauthorized`, репозиторий должен вернуть `Result.failure(UnauthorizedException())`. Не удалять токен внутри репозитория – это решение принимает `ViewModel`.

#### 3. `data.storage` — локальное хранилище

Класс `AuthStorage` (на базе `SharedPreferences` или `DataStore`):

* Хранит `authToken: String?` и `userName: String?`.
* Методы: `saveToken(token, userName)`, `getToken()`, `getUserName()`, `clear()`.
* Никакой бизнес-логики, только чтение/запись.

### Требования к модулю data

* Не содержит Android UI-зависимостей (`Context` только для доступа к `SharedPreferences`, но это допустимо в `storage`).
* Все публичные классы неизменяемы (`val`).
* Сетевые вызовы не блокируют поток (`suspend`).
* Отсутствуют `var` и мутабельные коллекции в публичных API.
* Нет хардкода строк (URL сервера вынести в константу, желательно в `build.gradle` или `BuildConfig`).

## Модуль `viewmodel`

**Назначение:** управление состоянием экранов, реакция на действия пользователя, вызов репозитория, преобразование DTO в UI-модели. Живёт дольше, чем экран (переживает повороты).

### Общие принципы
- Каждая ViewModel наследует `ViewModel()` из `androidx.lifecycle`.
- Состояние – `StateFlow<UiState>` (или `MutableStateFlow` внутри, наружу `StateFlow`).  
- Однократные события (навигация, диалоги) – `SharedFlow<Event>` или отдельное поле в `UiState`, которое UI сбрасывает.
- Все поля в `UiState` – `val`, коллекции – неизменяемые (`List`, `Set`).
- При получении `UnauthorizedException` от репозитория – очистить `AuthStorage` и отправить событие `navigateToLogin`.
- При сетевых ошибках – выставить `errorMessage` в состоянии.

### Конкретные ViewModel

#### 1. `LoginViewModel`
**Состояние `LoginUiState`:**
```kotlin
data class LoginUiState(
    val username: String = "",
    val password: String = "",
    val isLoading: Boolean = false,
    val errorMessage: String? = null
)
```
**События (однократные):** `sealed class LoginEvent { object NavigateToChatList : LoginEvent() }`

**Действия:**
- `updateUsername(String)`, `updatePassword(String)`
- `performLogin()` – вызывает `repository.login()`, при успехе сохраняет креды (через репозиторий), посылает событие `NavigateToChatList`. При ошибке – `errorMessage`.

#### 2. `ChatListViewModel`
**Состояние:**
```kotlin
data class ChatListUiState(
    val channels: List<String> = emptyList(),
    val isLoading: Boolean = false,
    val error: String? = null
)
```
**События:** `sealed class ChatListEvent { object NavigateToLogin : ChatListEvent() }`

**Действия:**
- `loadChannels()` – загружает список каналов.
- `selectChannel(channelId: String)` – сохраняет выбранный канал в `SharedChatViewModel`.
- `logout()` – вызывает `repository.logout()`, очищает `AuthStorage`, генерирует `NavigateToLogin`.

#### 3. `MessageViewModel`
Принимает `channelName: String` в конструкторе (через фабрику или `SavedStateHandle`).

**Состояние:**
```kotlin
data class MessageUiState(
    val messages: List<MessageUi> = emptyList(),
    val isLoading: Boolean = false,
    val isLoadingMore: Boolean = false,
    val hasMore: Boolean = true,      // есть ли ещё сообщения для пагинации
    val lastKnownId: Int = 0,         // последний полученный id, для следующей страницы
    val sendText: String = "",
    val error: String? = null
)
```
**MessageUi** – упрощённая модель для UI:
```kotlin
sealed class MessageUi {
    data class Text(val from: String, val text: String, val time: Long) : MessageUi()
    data class Image(val from: String, val thumbUrl: String, val fullUrl: String, val time: Long) : MessageUi()
}
```

**Действия:**
- `loadInitialMessages()` – загружает первые 20 сообщений (limit=20, lastKnownId=0). Формирует список, вычисляет `lastKnownId` (минимальный id? смотрим API: `lastKnownId` – с какого id начинать. Для следующей страницы нужно передавать последний полученный id. Допустим, сервер возвращает сообщения по убыванию id. Пагинация: первый раз lastKnownId=0, получаем id=100..80. Затем lastKnownId=80, получаем 79..60 и т.д.)
- `loadMoreMessages()` – если не загружается и `hasMore`, грузит следующую страницу с текущим `lastKnownId`.
- `sendTextMessage()` – отправляет текст, при успехе либо добавляем сообщение в начало списка (если сервер вернёт его с id), либо перезагружаем первые страницы (проще сделать `loadInitialMessages()` после отправки).
- `sendImageMessage(file)` – аналогично, через multipart.
- `updateSendText(String)`

При `UnauthorizedException` – событие `NavigateToLogin`.

#### 4. `SharedChatViewModel` (Activity-scoped)
```kotlin
data class SharedUiState(
    val selectedChannelId: String? = null
)
```
**Действие:** `selectChannel(String?)`

Используется только в ландшафтном режиме для синхронизации выбранного чата между списком и областью сообщений.

### Требования к модулю `viewmodel`
- Нет прямых ссылок на Android View или Context (кроме `SavedStateHandle` при необходимости).
- Нет мутабельных свойств, кроме `_state` и `_event` внутри.
- Все сетевые вызовы обрабатывают исключения и преобразуют их в состояния ошибок.
- Никаких `GlobalScope` – использовать `viewModelScope`.

## Модуль `ui`

**Назначение:** отображение интерфейса на Jetpack Compose, обработка жестов, системной кнопки «назад», навигация между экранами, адаптация к портретной/ландшафтной ориентации.

### Структура

- `MainActivity.kt` – точка входа, настраивает `NavHost`, определяет текущую ориентацию, создаёт `SharedChatViewModel` (через `viewModel()` с областью видимости Activity).
- `ui.navigation` – маршруты (sealed class `Screen`), `NavGraph`.
- `ui.screens` – отдельные композаблы:
  - `LoginScreen`
  - `ChatListScreen`
  - `MessageScreen`
  - `FullImageScreen`
- `ui.components` – переиспользуемые элементы (кнопка выхода, диалог ошибок, поле ввода, сообщение и т.д.)

### Навигация и маршруты
```kotlin
sealed class Screen(val route: String) {
    object Login : Screen("login")
    object ChatList : Screen("chatList")
    data class Messages(val channelId: String) : Screen("messages/{channelId}")
    data class FullImage(val imageUrl: String) : Screen("fullImage/{imageUrl}")
}
```

### Обработка ориентации и двухпанельный режим

В `MainActivity` или корневом композабле:
```kotlin
val configuration = LocalConfiguration.current
val isLandscape = configuration.orientation == Configuration.ORIENTATION_LANDSCAPE
if (isLandscape) {
    LandscapeApp(sharedViewModel, ...)
} else {
    PortraitApp(navController, ...)
}
```

**LandscapeApp:**
- Слева – `ChatListScreen` (с возможностью выбора канала). Выделение текущего выбранного чата.
- Справа – если `selectedChannelId != null`, показываем `MessageScreen` (с этим же `channelId`), иначе `PlaceholderScreen("Выберите чат")`.
- Кнопка «назад»: если `selectedChannelId != null` → очистить его (`sharedViewModel.selectChannel(null)`), иначе закрыть приложение.

**PortraitApp:**
- Обычный `NavHost` с навигацией между экранами.
- Кнопка «назад»:
  - На `MessageScreen` → вернуться на `ChatListScreen`.
  - На `ChatListScreen` → закрыть приложение.
  - На `FullImageScreen` → закрыть этот экран.
  - На `LoginScreen` → закрыть приложение.

**Реализация BackHandler:** в каждом экране (или общая логика в `NavHost`). Использовать `BackHandler(enabled = true) { ... }`.

### Описание экранов

#### `LoginScreen`
- Поля ввода username и password (TextField).
- Кнопка «Вход» – показывает `CircularProgressIndicator` при `isLoading`.
- При `errorMessage != null` – показать `AlertDialog` с сообщением, который закрывается по кнопке.
- При событии навигации `NavigateToChatList` – `navController.navigate(Screen.ChatList.route)`.

#### `ChatListScreen`
- `LazyColumn` из каналов (простые текстовые элементы или карточки).
- По клику – в портрете навигировать на `Screen.Messages(channelId)`, в ландшафте – `sharedViewModel.selectChannel(channelId)`.
- Кнопка выхода (иконка в `TopAppBar`) вызывает `viewModel.logout()`.
- Обработка ошибок: при `error` показывать Snackbar.
- При событии `NavigateToLogin` – очистить стек и перейти на `LoginScreen`.

#### `MessageScreen`
- Параметр `channelId` получает из аргументов навигации (в портрете) или из `SharedChatViewModel` (в ландшафте).
- Отображение сообщений в `LazyColumn`:
  - `TextMessage` – текст.
  - `ImageMessage` – `AsyncImage` с `thumbUrl`. По клику – навигация на `FullImageScreen` с `fullUrl`.
- Пагинация: при достижении конца списка (например, `LazyListState.lastVisibleItemIndex == messages.size - 1` и `!isLoadingMore && hasMore`) вызывать `viewModel.loadMoreMessages()`.
- Нижнее поле ввода: `TextField` + `Button` (отправка только непустого текста). 
- Для отправки картинок можно добавить отдельную иконку (выбор файла через `rememberLauncherForActivityResult(ActivityResultContracts.GetContent())`). Конвертировать URI в `MultipartBody.Part` и вызвать `viewModel.sendImageMessage(...)`.
- Кнопка перехода на экран чатов (иконка «назад» или стрелка в `TopAppBar`) – в портрете `navController.popBackStack()`.

#### `FullImageScreen`
- Аргумент `imageUrl`.
- `AsyncImage` на весь экран с `Modifier.fillMaxSize()` и `contentScale = ContentScale.Fit`.
- Кнопка закрытия (крестик сверху) или `BackHandler` – закрыть экран (popBackStack).

### Общие требования к UI
- Все строки вынесены в `res/values/strings.xml`.
- Не использовать `@Preview` аннотации для продакшн-кода? Можно, но не обязательно.
- При повороте не должно быть повторных вызовов `loadChannels` / `loadMessages` (благодаря ViewModel).
- Открытый чат сохраняется: в портрете – через аргументы навигации, в ландшафте – через `SharedChatViewModel`.
- Картинки загружаются через Coil (библиотека `io.coil-kt:coil-compose`), кеширование автоматическое.

### Требования к модулю `ui`
- Никакой бизнес-логики – только вызовы методов ViewModel и навигация.
- Нет использования `var` для состояния, все подписки через `collectAsState()`.
- Нет хардкода строк и цветов (цвета – через `MaterialTheme` или отдельные ресурсы).

## Модуль `utils`

**Назначение:** вспомогательные функции, расширения, константы, не зависящие от бизнес-логики, используемые в `ui` и `viewmodel`.

### Конкретные утилиты

#### 1. Обработка ошибок
```kotlin
fun Throwable.toUserMessage(context: Context): String {
    return when (this) {
        is UnauthorizedException -> context.getString(R.string.error_unauthorized)
        is HttpException -> when (code()) {
            404 -> context.getString(R.string.error_not_found)
            else -> context.getString(R.string.error_network_general)
        }
        is IOException -> context.getString(R.string.error_no_internet)
        else -> context.getString(R.string.error_unknown)
    }
}
```
Но лучше передавать строки как параметр, чтобы избежать зависимости от `Context` в `utils`. Можно сделать `toUserMessage( resourceProvider: (Int) -> String )`.

#### 2. Форматирование времени
```kotlin
object DateFormatUtils {
    fun formatTime(timestamp: Long): String {
        // SimpleDateFormat с учётом локали
    }
}
```

#### 3. Валидация
```kotlin
fun isValidUsername(input: String): Boolean = input.isNotBlank()
fun isValidPassword(input: String): Boolean = input.isNotBlank()
```

#### 4. Расширения для коллекций (безопасная пагинация)
```kotlin
fun <T> List<T>.safeSubList(fromIndex: Int, toIndex: Int): List<T> {
    // обрезает по реальным границам
}
```

#### 5. Константы
```kotlin
const val BASE_URL = "https://faerytea.name/"
const val DEFAULT_CHANNEL = "1@channel"
const val MESSAGES_PAGE_SIZE = 20
```

### Требования к модулю `utils`
- **Абсолютно никаких зависимостей от Android** (кроме `Context` только если явно не избежать, но лучше использовать провайдер ресурсов).
- Все функции pure (без побочных эффектов, без изменения переданных объектов).
- Нет состояния (никаких `object` с mutable внутренностями).
- Может быть использован любым другим модулем без циклических ссылок.

## Взаимодействие модулей (зависимости)

- `ui` → `viewmodel` (использует ViewModel от AndroidX) → `utils`  
- `ui` → `utils` (для форматирования, утилит)  
- `viewmodel` → `data` (репозиторий)  
- `viewmodel` → `utils` (обработка ошибок, константы)  
- `data` → `utils` (только если нужны общие утилиты, но лучше минимально)  

**Запрещены циклы:** например, `data` не знает о `viewmodel`, `viewmodel` не знает о `ui`.
