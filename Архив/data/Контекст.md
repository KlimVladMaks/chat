Привет! Я сейчас изучаю в университете дисциплину "Мобильная разработка на Android" и мне дали учебное задание по разработке мобильного приложения чата по следующему ТЗ:
"""
# Чат

## Экраны

В приложении ожидаются следующие экраны:

### Экран входа

- Поле для ввода логина
- Поле для ввода пароля
- Кнопка входа
- Адекватная реакция на ошибки (не '401: Unauthorized' в логи, а 'Неверный логин / пароль' в диалоговом окне)
- Пропускается, если логин и пароль уже вводились

### Экран чатов (портрет)

- Список чатов
  - Каналы (см `/channels`)
  - Личные сообщения (будут появляться в `/inbox`) пока реализовывать не нужно
  - Нажатие на чат открывает его
- Кнопка выхода (ведёт на экран входа)

### Экран сообщений (портрет)

- Подгружает (и показывает) сообщения из чата
  - Текстовые как есть
  - Картинки из `/thumb/`, по клику открывает на весь экран
  - Пагинация: загружать не более 20 сообщений за раз
- Поле для ввода текста
- Кнопка отправки сообщения (отправляет непустое сообщение)
- Кнопка перехода на экран чатов

### Экран картинки

- Картинка из `/img/` на весь экран
- Адекватный способ её закрыть (переходя на экран чата)

### Экран чатов и сообщений (ландшафт)

- Слева содержимое экрана чатов
- Справа надпись "выберите чат" или содержимое экрана сообщений
- Открытый чат должен быть как-то обозначен в списке

## Общее

- При повороте должен сохраняться открытый чат (даже если во время поворота открыта картинка)
- Кнопка (при трёхкнопочном интерфейсе) и жест (при жестовом) "назад" должны вести себя ожидаемо
  - Не надо разлогинивать из списка чатов
  - На ландшафте кнопка должна закрывать чат, если есть
  - На портрете переходить с экрана сообщений на экран чатов
  - На картинке закрывать картинку
  - В остальных случаях закрывать приложение
- Поворот не должен вызывать поход в сеть
- 401: Unauthorized всегда запускает процесс входа заново
  - Так как сервер может работать нестабильно и токен может слетать, то крайне желательно сохранять логин+пароль на устройстве; это плохо в реальном приложении, но в рамках данного учебного приложения приемлемо
- Ожидается использование каких-нибудь адекватных библиотек для работы с сетью
  - Volley и голый java.net к ним не относятся, системный Apache HTTP Client тем более (особенно учитывая, что он удалён).

## Нежелательно

- Любой warning студии (жёлтое подчёркивание)
- Использование мутабельных (или вероятно мутабельных) объектов для состояния

## Всё плохо

- Пакет приложения начинается с `com.example`
- Не собирается
- Падает
- Заблокирован поворот в манифесте
- Хардкод строк
  - Alt+Enter → Extract string resource…

---

# Дополнительная информация

* При разработке используется подход Jetpack Compose.
* Это учебный проект, так что он должен быть максимально простым и понятным, но при этом соответствовать описанному выше заданию.
"""

Я решил разделить проект на следующие пакеты:

* `data` - отвечает за работу с данными (по большей части работает с сервером через API).
* `viewmodel` - содержит ViewModel-модели.
* `ui` - содержит экраны, компоненты и т.п.
* при необходимости может быть создан пакет `utils`, если потребуются какие-то дополнительные инструменты.

Давай сейчас поработаем над пакетом `data`. Для данного пакета у меня есть следующее ТЗ для работы с API:
"""
# Как тут ходить

Сервер лежит по адресу https://faerytea.name/

## Регистрация

-   Web:  /adduser.html. Написать имя пользователя, получить пароль, запомнить.
-   API: POST на  `/addusr`, в  **теле**  запроса  `name=<my name>`.  `<my name>`  заменить на имя пользователя. Ответом придёт plain text вида  `password: '<password>'`.

## Логин

POST на  `/login`  JSON следующего вида:

```json
{ "name": "<имя пользователя>", "pwd": "<пароль>" }
```

В ответ придёт токен. Токен кладётся в заголовок  `X-Auth-Token`. Токен живет до рестарта сервера.

Токен просто приходит в формате "text/plain; charset=UTF-8". Например, вот так:
```
H+/OOSZ7Qr2S05A2AAABngNL260=
```

Куда нужен токен:

-   GET  `/inbox/<username>`
-   POST  `/messages`
-   POST  `/logout`

## Что тут бывает

### Сообщение

```json
{
  "id": "123", // игнорируется в запросах, обязательно будет в ответе
  "from": "my name", // обязательное, "от кого", должно совпадать с именем пользователя
  "to": "1@channel", // опциональное в запросах, по умолчанию "1@channel"
  "data": {
    "Text": {
      "text": "my message" // обязательно
    },
    // ИЛИ (исключающее)
    "Image": {
      "link": "path/to/pic.jpg" // опционально для запросов
    }
  },
  "time": "1297490" // UNIX time, опционально для запросов
}

```

### Списки

```json
["elem1", "elem2", "elem3"]
```

-   Список каналов
-   Список пользователей
-   Список печатающих

## Куда и как ходить

### Получение сообщений

-   GET  `/1ch`  
    Список сообщений из канала  `1@channel`, то же самое, что и  `/channel/1@channel`
-   GET  `/channel/<имя канала>`  
    Список сообщений из указанного канала
-   GET  `/inbox/<имя пользователя>`  
    Список входящих и исходящих сообщений конкретного пользователя

Параметры запросов:

-   `limit`  
    Сколько сообщений отдавать, по умолчанию 20
-   `lastKnownId`  
    Начиная с какого сообщения отдавать сообщения, по умолчанию 0
-   `reverse`  
    Возвращает сообщения в обратном порядке (с  `id`  _меньше_, чем  `lastKnownId`), по умолчанию  `false`

### Получение информации

-   GET  `/channels`  
    Список каналов в виде JSON массива строк
-   GET  `/users`  
    Список  _зарегистрированных_  пользователей в виде JSON массива строк
-   GET  `/img/<path>`  
    Картиночка по указанному пути в хорошем разрешении
-   GET  `/thumb/<path>`  
    Картиночка по указанному пути в шакальном разрешении
-   GET  `/typing/<channel>`  
    Кто тут печатает

### Отправка информации

-   POST  `/1ch`  (или  `/messages`) с сообщением в body и  `Content-Type=application/json`
    -   `200 OK`  и  `id`  поста
    -   Ошибку, смотри коды состояния HTTP
-   POST  `/1ch`  (или  `/messages`) с сообщением в body и  `Content-Type=multipart/form-data`
    -   `200 OK`  и  `id`  поста
    -   Ошибку, смотри коды состояния HTTP
    -   Сообщение идёт с названием  `msg`
    -   Картинка идёт с названием  `picture`
    -   Используйте  `@Multipart`  из Retrofit
-   POST  `/logout`  
    Забывает токен, всегда успешно

## А как сделать чат?

Просто напишите в него!

Таким же образом можно сделать канал, написав в  `<что угодно>@channel`
"""

Для пакета `data` я подготовил следующую структуру:
```
data:
    api:
        dto:
            LoginRequest.kt
            Message.kt
            MessageData.kt
            # Все остальные данные возвращаются в виде строк или списков, так что для них dto я не делал
        ApiInterface.kt
        RetrofitClient.kt
    repository:
        ChatRepository.kt
    storage:
        AuthStorage.kt
```

- okhttp3
- retrofit
- и другие необходимые технологии
