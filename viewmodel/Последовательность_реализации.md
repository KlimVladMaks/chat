## Рекомендуемый порядок реализации ViewModel

Логика разработки — идти от **базового и независимого** к **более сложному и зависимому**, постоянно имея работающее приложение после каждого этапа.

### Этап 1. `MainViewModel` (самый важный, без него ничего не работает)

Почему сначала?
- Он хранит глобальное состояние: аутентификация, выбранный канал, полноэкранное изображение.
- Он управляет обработкой `UnauthorizedException` и логикой выхода.
- Все остальные ViewModel будут получать ссылку на `MainViewModel` (или вызывать его методы через колбэки/события).

Что реализовать:
- Поля `_state` и `state`
- `checkAuth()` — читает токен из `AuthStorage`
- `logout()` и `onUnauthorized()`
- `selectChannel()`, `showImage()`, `hideImage()`
- Пустые заглушки методов, чтобы их можно было вызывать из других ViewModel (пока не будет UI, можно не реализовывать сложную логику, но сигнатуры должны быть).

Проверка: после создания `MainViewModel` (без UI) можно написать простой тест, что методы корректно обновляют `StateFlow`.

### Этап 2. `LoginViewModel`

Почему вторым?
- Он не зависит от других ViewModel, только от `MainViewModel` (вызов `mainViewModel.checkAuth()` после успешного входа).
- С его помощью можно реализовать экран входа и убедиться, что аутентификация работает.

Что реализовать:
- Поля состояния: `username`, `password`, `isLoading`, `errorMessage`.
- `updateUsername()`, `updatePassword()`
- `login(mainViewModel, onSuccess)` — вызывает `repository.login()`, при успехе обновляет `MainViewModel`, при ошибке выставляет `errorMessage`.

Проверка: подключить `LoginViewModel` к Compose-экрану входа (без других экранов), проверить:
- Ввод логина/пароля
- Ошибка «Неверный логин/пароль» при 401
- После успешного входа `MainViewModel.isAuthenticated == true`.

### Этап 3. `ChannelsViewModel`

Почему третьим?
- Для отображения списка чатов нужна авторизация (которая уже есть).
- Загружает каналы через `repository.getChannels()`.
- Взаимодействует с `MainViewModel` через `selectChannel()`.

Что реализовать:
- Состояние: `channels`, `isLoading`, `error`
- `loadChannels(mainViewModel)` — загружает один раз (кеширует)
- `onChannelClick(channel, mainViewModel, navController, isLandscape)` — вызывает `mainViewModel.selectChannel()` и при портрете навигацию.

Проверка:
- После входа открывается список каналов.
- При клике на канал в портрете открывается экран чата (пустой пока).
- В ландшафте просто меняется `selectedChannel` в `MainViewModel` (можно вывести логи или временный toast).

### Этап 4. `ChatViewModel`

Почему четвёртым?
- Самый сложный: пагинация, отправка сообщений, загрузка картинок (пока только ссылки на `/thumb/`).
- Зависит от выбранного канала из `MainViewModel` (или `SavedStateHandle`).

Что реализовать:
- Состояние: `messages`, `isLoadingInitial`, `isLoadingMore`, `hasMore`, `sendText`, `isSending`, `error`
- `loadInitialMessages(mainViewModel)` — загружает 20 сообщений с сервера
- `loadMoreMessages(mainViewModel)` — пагинация
- `updateSendText()`
- `sendMessage(mainViewModel)` — отправка текста, затем перезагрузка сообщений
- При изменении канала (через `savedStateHandle` или наблюдение за `MainViewModel.selectedChannel`) — перезагружать сообщения.

Проверка:
- Открывается чат, видны сообщения (текстовые и ссылки на картинки).
- Прокрутка вверх подгружает следующие сообщения.
- Можно отправить текст, он появляется в списке после отправки.
- Обработка ошибок и 401.

### Этап 5. `ImageViewModel` (опционально, можно и без него)

Почему последним?
- Относительно простой, нужен только для полноэкранного просмотра картинок.
- Можно реализовать прямо в Compose без отдельной ViewModel, но для чистоты архитектуры и переиспользования — выделить отдельно.

Что реализовать:
- Состояние: `isLoading`, `bitmap`, `error`
- `loadImage(repository, path, mainViewModel)` — загружает полное изображение по пути из `MessageData.Image.link`.

Проверка:
- Клик на миниатюру (из сообщения) открывает полноэкранное изображение.
- Кнопка «Назад» закрывает картинку (через `mainViewModel.hideImage()`).

## Дополнительные советы по реализации

### Итеративный подход
- После каждого этапа **убеждайся, что приложение компилируется и не падает**.
- Используй заглушки для ещё не реализованного UI (например, вместо экрана чата показывай `Text("Chat: $channel")`).
- Постепенно добавляй функциональность, не пытаясь реализовать всё сразу.

### Работа с `MainViewModel` в других ViewModel
Чтобы избежать циклических зависимостей и упростить тестирование, передавай в `LoginViewModel`, `ChannelsViewModel` не сам `MainViewModel`, а **интерфейс** с методами `onUnauthorized()` и `selectChannel()`, или используй `Channel`/`SharedFlow` для событий. Для учебного проекта допустимо передавать `MainViewModel` напрямую — это наглядно и просто.

### Упрощение `ChatViewModel` в ландшафте
В двухпанельном режиме `ChatViewModel` должен жить в `Activity` (или быть привязан к `MainViewModel` с ключом по каналу), чтобы при повороте не пересоздаваться. Но на начальном этапе можно сделать так, чтобы в ландшафте просто пересоздавался `ChatViewModel` при смене канала — это не страшно, главное, чтобы сообщения перезагружались. Для соответствия ТЗ «поворот не должен вызывать поход в сеть» нужно, чтобы `ChatViewModel` для каждого канала сохранялся. Самый простой способ: хранить `Map<String, ChatViewModel>` внутри `MainViewModel` или использовать `ViewModelStore` с ключами. Но это усложнение. Для начала **можно пренебречь** идеальным поведением при повороте в ландшафте, если ты ещё не дошел до реализации двухпанельного режима. Позже легко доработать.

### Порядок тестирования UI (Compose)
- На этапе 2: экран входа.
- Этап 3: экран чатов.
- Этап 4: экран сообщений.
- После этапа 4 уже есть полноценное приложение (без полноэкранных картинок, но с миниатюрами).
- Этап 5: доработка просмотра картинок.

### Резюме: конкретная последовательность шагов

1. **Создать классы `MainState`, `MainViewModel`** (с заглушками).
2. **Создать `LoginState`, `LoginViewModel`** и экран входа на Compose.
3. **Добавить навигацию, которая переключается на экран чатов после входа** (экран чатов пока пустой).
4. **Создать `ChannelsState`, `ChannelsViewModel`** и экран списка каналов.
5. **Реализовать экран сообщений с `ChatState`, `ChatViewModel`** (без пагинации, сначала просто загрузить 20 сообщений и отправить текст).
6. **Добавить пагинацию** (подгрузка при прокрутке вверх).
7. **Добавить отображение картинок** (миниатюры через `AsyncImage` или `Coil` с `/thumb/`).
8. **Создать `ImageViewModel`** и полноэкранный просмотр.
9. **Реализовать двухпанельный ландшафтный режим** и сохранять `selectedChannel` при повороте.
10. **Добавить обработку системной кнопки «Назад»** через `BackHandler` на всех экранах.

Такая последовательность даёт работающее приложение после каждого шага и не перегружает голову сразу всей сложностью. Удачи с проектом!
